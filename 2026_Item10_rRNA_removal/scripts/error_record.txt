这个问题很典型！这是因为 Nextflow 的 resume 机制导致的。

问题根源
你在命令中使用了 -resume 参数：

bash
-resume
当你第一次运行错误脚本时，Nextflow 创建了一个工作会话（session），并记录了会话信息在 .nextflow/ 目录中。

当你修改参数后再次运行，即使参数已经正确，但因为使用了 -resume，Nextflow 会：

查找之前的会话：根据相同的流程名称和参数哈希值
复用之前的配置：包括工作目录路径、输入输出路径等
继续之前的执行：而不是开始一个全新的运行
这就是为什么你看到的日志仍然是旧的错误参数！

解决方案
方案1：删除 Nextflow 缓存（推荐）
删除 .nextflow/ 目录，强制 Nextflow 开始全新运行：

bash
# 在 scripts 目录下执行
rm -rf .nextflow/
# 然后再运行你的脚本
bash command_final_for_rRNA.sh
方案2：临时移除 -resume 参数
先不使用 -resume 运行一次，确认一切正常后再加回来：

bash
# 修改脚本，暂时注释掉 -resume 行
# -resume
运行成功后再恢复 -resume 参数。

方案3：使用不同的运行名称
通过 -name 参数指定不同的运行名称：

bash
nextflow run nf-core/rnaseq \
    -name item10_correct_run \  # 添加这一行
    -r 3.15.1 \
    # ... 其他参数保持不变 ...
    -resume
验证步骤
删除 .nextflow/ 目录
运行修正后的脚本
检查新的 .nextflow.log 文件，确认参数是否正确
确认 work 目录是否在正确位置创建
这样就能确保你运行的是当前正确的脚本参数了！
